# Анализ утечек памяти (Спринт 7)

Документ описывает процесс проверки утечек памяти и правила разработки для спринта 7.

Файл используется для фиксации результатов анализа утечек памяти по спринту 7. Итоговый прогон делается после того, как будут готовы и смержены задачи **1–5**.

---

## 1. План проверки (что делаем в конце спринта)

### 1.1. Клиентская часть

- **Сценарии**
  - Многократные переходы между страницами (роуты React Router).
  - Повторный запуск игры несколько раз подряд.
  - Логин / логаут / OAuth‑авторизация.
  - Открытие и закрытие модалок / попапов (если есть).

- **Критерии утечки:**
  - Heap size растёт после каждого цикла и не возвращается к исходному значению (допустимо ±5–10%)
  - Количество DOM‑узлов монотонно увеличивается
  - Количество event listeners увеличивается после каждого цикла

- **Инструменты DevTools**
  - Вкладка **Performance**:
    - Записываем профиль при нескольких переходах между страницами и повторах игры.
    - Смотрим, не растёт ли монотонно количество DOM‑узлов и event listeners.
  - Вкладка **Memory**:
    - **Heap snapshot** по методологии:
      1. Snapshot после загрузки страницы (baseline)
      2. Snapshot после 10 циклов «запуск игры → выход»
    - Сравниваем retained objects и detached DOM nodes между снапшотами.
    - Проверяем наличие **Detached DOM nodes** в Heap Snapshot — классический симптом утечек в React.

### 1.2. Серверная часть (SSR)

- **Сценарии**
  - Многоразовые HTTP‑запросы к одному и тому же роуту (особенно там, где создаётся Redux store).
  - Переходы по разным SSR‑роутам подряд.

- **Что проверяем**
  - На каждый запрос создаётся **новый Redux store**, нет повторного использования одного и того же store между запросами.
  - Нет **глобальных изменяемых переменных**, в которых накапливается состояние между запросами (кроме явных кешей, оформленных аккуратно).
  - Нет «висящих» таймеров / интервалов / подписок, которые создаются на каждый запрос и не очищаются.

- **Инструменты для Node.js (при необходимости)**
  - `--inspect` + Chrome DevTools (Memory-вкладка подключается к Node.js процессу)
  - `process.memoryUsage()` — логирование RSS / heapUsed между запросами для выявления роста

## 2. Правила для команды по ходу разработки задач 1–5

### 2.1. Общие правила (клиент и SSR)

- **Слушатели и подписки**
  - Любой `addEventListener`, `subscribe`, `observer`, `WebSocket`, `setInterval`, `setTimeout` — **обязан иметь парный cleanup**:
    - В React‑компонентах — через `useEffect(() => { ...; return () => { cleanup } }, [...])`.
    - В «голом» коде — явно вызывать `removeEventListener`, `unsubscribe`, `clearInterval`, `clearTimeout` при завершении работы.

- **Работа с внешними ресурсами**
  - Не хранить крупные объекты (большие массивы, результаты API, Canvas и т.п.) в замыканиях, которые живут дольше, чем нужно.
  - Не складывать временные данные в глобальные синглтоны, если их можно держать в локальном состоянии или в Redux.

### 2.2. Для задачи 1: Настроить Express для SSR (без Redux)

- **Что учесть**
  - Не хранить `req`/`res` или большие объекты запроса в глобальных массивах/кешах без необходимости.
  - Если где‑то создаются таймеры/интервалы (например, для логирования/метрик), убедиться, что они создаются **один раз при старте сервера**, а не при каждом запросе.

### 2.3. Для задачи 2: Настроить Redux и Router в SSR

- **Store per request**
  - Функция создания store должна выглядеть как `createStore(initialState)` и вызываться **на каждый HTTP‑запрос**.
  - Нельзя держать один общий `store` в модуле и переиспользовать его между разными запросами.

- **Router**
  - В SSR использовать `StaticRouter` с `location` из запроса.
  - Не хранить `StaticRouter` или связанные с ним объекты в глобальном состоянии.

### 2.4. Для задачи 3: Добавить OAuth‑авторизацию (уже сделано)

- **Что уже проверили / договорились**
  - При обработке `code` в query‑параметрах нет бесконечной перерисовки (эффект в `Layout` должен отрабатывать один раз).
  - После завершения OAuth‑процесса важно, чтобы состояние, связанное с запросом, не копилось в глобальных структурах.

### 2.5. Для задачи 4: Добавить API для лидерборда

- **API‑клиент**
  - Если используется какой‑то клиент/обёртка для запросов (например, инстанс `fetch`/`axios`), он может быть синглтоном, **но он не должен накапливать в себе состояние** (коллекции активных запросов, глобальный кэш и т.п.), если это не продуманная фича.

- **Компоненты**
  - При отображении таблицы лидеров избегаем бесконечных подписок/таймеров для автообновления. Если нужны периодические опросы, делаем `setInterval` + `clearInterval` в `useEffect`.

### 2.6. Для задачи 5: Дополнительное Web API

- Любое Web API (Fullscreen, Geolocation, Performance, Notification и т.д.) часто вешает свои обработчики.
  - Убедиться, что:
    - При выходе со страницы или размонтировании компонента все обработчики и подписки снимаются.
    - Для одноразовых операций (например, единичный запрос геолокации) нет «вечных» слушателей.

## 6. Частые источники утечек

Типичные паттерны, которые приводят к утечкам памяти и которых нужно избегать:

- **`useEffect` без cleanup** — подписки, таймеры и слушатели, созданные в `useEffect`, должны сниматься в функции cleanup (`return () => { ... }`)
- **`setInterval` без `clearInterval`** — интервалы продолжают работать после размонтирования компонента
- **Подписки на store без `unsubscribe`** — например, `store.subscribe(...)` без сохранения и вызова возвращаемой функции отписки
- **Хранение данных в глобальных переменных** — объекты, добавленные в глобальный скоуп, никогда не удаляются сборщиком мусора
- **Повторное использование Redux store в SSR** — один общий store между запросами приводит к накоплению состояния и потенциальной утечке данных между пользователями
